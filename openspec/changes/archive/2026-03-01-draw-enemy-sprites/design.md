## Context

The dungeon has a tile-based rendering system using a sprite atlas (`dungeon-tileset.png`, 32x32 tiles). The player, items, and map tiles are already rendered via `SimpleTileRenderer` with atlas coordinates defined in `DungeonTileset`. Rooms are procedurally generated by `DungeonGenerator`, and per-level state (map, items, player position) is cached in `GameState.dungeonCache`.

Enemies need to follow the same patterns: define sprites in the tileset, store enemy data per level, spawn during generation, and render in the scene.

## Goals / Non-Goals

**Goals:**
- Define 3 enemy types with distinct sprites from the existing atlas (skeleton, slime, bat)
- Spawn enemies in dungeon rooms during generation
- Render enemy sprites on the map (between items and player in draw order)
- Cache enemies per level so they persist when backtracking

**Non-Goals:**
- Enemy movement or AI (enemies are static for now)
- Combat system or damage dealing
- Enemy health bars or status indicators
- Enemy animations

## Decisions

### Enemy data model: simple data class with position and type
Use a flat `Enemy` data class with `type: EnemyType`, `x: Int`, `y: Int` — matching the `Item` pattern. Stats (hp, attack, defense) are included per-enemy to support future combat, but are not used yet.

**Alternative considered:** An entity-component system would be more flexible but is over-engineered for static enemies with no behavior yet.

### Spawning: random placement in rooms, avoiding occupied tiles
Enemies spawn during `DungeonGenerator.generate()`, similar to item placement. Each room (except the starting room) has a chance to contain 1-3 enemies. Placement avoids walls, doors, stairs, items, and other enemies.

**Alternative considered:** Fixed enemy placement per room template — rejected because we don't have room templates yet.

### Enemy type selection: level-scaled
Slimes appear on all levels, skeletons from level 2+, bats from level 3+. This provides a simple difficulty progression.

### Sprite coordinates: use existing atlas characters
The `dungeon-tileset.png` atlas has character sprites in the top rows. We'll select 3 visually distinct sprites for the enemy types.

## Risks / Trade-offs

- **[Static enemies feel lifeless]** → Acceptable for this phase; movement/AI is a natural follow-up change
- **[Enemies block movement paths]** → Enemies are placed on floor tiles but don't block movement yet (player walks through them). Combat will handle collision later
- **[Atlas coordinate brittleness]** → Same risk as existing tile definitions; mitigated by centralizing all coordinates in `DungeonTileset`
